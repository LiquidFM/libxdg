Binary cache of Desktop Entry Specification.

The specification could be found here.

This specification describes directory layout and format of ".desktop" and ".list" files. Purpose of the library is to provide fastest possible access to contents of this files. This goal is achieved by using of AVL trees. They were chosen because AVL trees are more rigidly balanced than red-black trees, leading to slower insertion and removal but faster retrieval.

Format of binary cache which contains data from ".desktop" and ".list" files is:

  • 4 bytes for version of a cache file.

  • List of XdgFileWatcher structures (at least one zeroed structure, i.e NULL-terminated C list).
    This list is used during loading of cache to determine validity of cache.
    See also:
    xdg_app_cache_file_is_valid()
    xdg_app_refresh()
  • AVL tree of all ".desktop" files located in the same directory (and subdirectories) as cache file.
    Each key of this tree is a name of ".desktop" file (e.g. kde4-kate.desktop).
    Each value of this tree is an AVL tree of XdgApp items, which in turns is an AVL tree of XdgAppGroup items, which is an AVL tree of XdgAppGroupEntry items.
    Note:
    This tree serves as container of ".desktop" files for other two major trees in cache.
  • AVL tree of associations of mime type with ".desktop" files.
    Each key of this tree is a name of a first part of mime type (e.g. text).
    Each value of this tree is an AVL tree of a second part of mime type (e.g. html), which contains a list of pointers to XdgApp items.
    Note:
    Global version of this tree is used to resolve ".desktop" files according to the given mime type.
    See also:
    xdg_known_apps_lookup()
  • AVL tree of all ".list" files located in the same directory (and subdirectories) as cache file.
    Each key of this tree is a name of a group from ".list" file (e.g. Default Applications).
    Each value of this tree is an AVL tree of entries, which in turns is an AVL tree of {mime type, ".desktop" files} pairs in the format:
    "text/plain=kde4-kate.desktop;diffuse.desktop;".
    Note:
    Global version of this tree is used to resolve ".desktop" files according to the given mime type.
    See also:
    xdg_default_apps_lookup()
    xdg_added_apps_lookup()
    xdg_removed_apps_lookup()


What are the Global AVL trees mentioned above? Well the thing is, or even the problem is that we have several directories with data (".desktop" and ".list" files) and separate cache file for each directory. This means that we could have reference to ".desktop" file which is located in a different directory. More of that, this ".desktop" file could be in a valid cache file of that directory...
Despite this we would like to have a global picture of the registered applications in the system. That's why we need the Global versions of this trees, which contains references to ".desktop" files in appropriate directory. Also this approach gives as a possibility to rebuild the cache only in one data directory.

Note:
Actually, this problem is solved a bit different way. There is no Global AVL trees, there is a list of directories with their data. When data loaded for each directory we are fixing references between data of this directories.
That's why XdgJointList appeared - to dynamically group (create references) data from different folders without memory allocations.
Also, this is the root of thread unsafety.
